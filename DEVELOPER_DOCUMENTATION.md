# **Документация разработчика**

# **1. Общие сведения о проекте**

Проект представляет собой мобильное приложение, предназначенное для оптимизации личных финансов студентов. Система позволяет вести учет доходом и расходов, устанавливать лимиты по категориям и получать аналитические отчеты.

# **2. Архитектура системы**

Приложение построено по модульному принципу для обеспечения гибкости в рамках итерационной разработки (Agile).

Обоснование выбора:

1. Гибкость требований: специфика мобильного приложения для студентов подразумевает возможные изменения в интерфейсе и функционале (например, добавление новых категорий или способов визуализации) на основе промежуточного тестирования.
2. Командная работа: в команде распределены четкие роли (менеджер, разработчик, аналитик, техписатель, дизайнер), что позволяет эффективно работать короткими спринтами.
3. Снижение рисков: в отличие от каскадной модели (Waterfall), итерационный подход позволяет выявлять ошибки реализации на ранних стадиях, а не только в конце проекта.
4. Частые релизы: использование Agile позволяет выпускать работающие модули (например, сначала модуль учета расходов, затем аналитику) постепенно.

Основные программные модули:

1. Модуль авторизации и защиты: обеспечивает безопасность пользовательских данных через PIN-код и биометрическую аутентификацию. Отвечает за регистрацию, вход в систему и управление доступом.
2. Модуль учета операций и расчета баланса (Accounting Core): обрабатывает все финансовые операции (доходы и расходы), связывает их с категориями и пользователем, автоматически рассчитывает текущий баланс на основе всех транзакций.
3. Модуль контроля лимитов (Budget Control): реализует логику установки и отслеживания лимитов по категориям, контролирует уровень расходов и определяет превышение заданных порогов (например, 80% от лимита).
4. Модуль генерации уведомлений: отвечает за создание и отправку уведомлений пользователю (о превышении лимитов, новых советах, изменениях баланса), а также хранение истории уведомлений и управление их статусом (прочитано/не прочитано).
5. Модуль аналитики и прогнозирования: выполняет обработку финансовых данных пользователя, формирует статистику (доходы, расходы, баланс), строит визуализации (графики, диаграммы) и может прогнозировать будущие расходы на основе предыдущих периодов.
6. Модуль работы с долгами: позволяет учитывать долги (кому должен пользователь или кто должен ему), отслеживать сроки возврата, статус (активный/погашен) и формировать напоминания о предстоящих платежах.
7. Модуль управления копилками: обеспечивает создание финансовых целей (копилок), отслеживание накоплений, расчет прогресса достижения цели и управление пополнением.
8. Модуль формирования советов: анализирует поведение пользователя (расходы, категории, превышения лимитов) и на основе этого генерирует персонализированные рекомендации по оптимизации финансов (например, снижение трат в определённых категориях).

# **3. Структура базы данных**

Для хранения данных используется реляционная СУБД PostgreSQL.

Основные сущности (Entities):

- User: id (PK), name, email, pin_hash, created_at.
- Notification: id (PK), user_id (FK), title, text, type, is_read, created_at.
- Advice: id (PK), user_id (FK), text, created_at.
- Category: id (PK), name, monthly_limit, user_id (FK).
- Transaction: id (PK), user_id (FK), category_id (FK), type (income/expense), amount, date, comment.
- Income: id (PK), operation_id (FK), amount, source, date.
- Expense: id (PK), operation_id (FK), amount, date, comment.
- Limit: id (PK), category_id (FK), monthly_value, alert_threshold.
- Analytics: id (PK), user_id (FK), period, total_income, total_expense, balance.
- Goal: id (PK), user_id (FK), title, target_amount, current_amount, deadline.
- Debt: id (PK), user_id (FK), person_name, amount, due_date, status, comment.

# **4. Описание программных интерфейсов (API)**

Документирование кода выполняется с использованием стандарта Doxygen для автоматической генерации технического описания.

Правила документирования:

- каждый класс должен сопровождаться тегом @class с описанием его назначения;
- публичные методы снабжаются описанием параметров (@param) и возвращаемых значений (@return);
- сложные алгоритмы (например, расчет прогноза бюджета) должны иметь комментарии внутри блоков кода;

# **5. План развертывания и тестирования**

1. Развертывание окружения: Клонирование репозитория, настройка зависимостей через пакетный менеджер (npm/pub).

3. Приемка: Проверка соответствия реализованных функций User Stories, утвержденным на этапе анализа.

# **6. Технологический стек**

Для работы над проектом установлены следующие требования:
